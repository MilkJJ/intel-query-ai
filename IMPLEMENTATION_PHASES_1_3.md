## Data Flow Architecture

```
User Query (frontend)
    ↓
POST /query (FastAPI endpoint)
    ↓
AgentRouter.handle(AgentContext)
    ↓
RouterAgent.route_query(context)
    ├→ IntentClassifier._classify_intent(query) → IntentType
    ├→ Match intent to agent(s)
    │
    ├─ TRANSCRIPTION:
    │  └→ TranscriptionAgent → TranscriptionMCP (gRPC) → audio extraction
    │
    ├─ VISION:
    │  └→ VisionAgent → VisionMCP (gRPC) → frame analysis
    │
    ├─ GENERATION:
    │  ├→ VisionAgent → VisionMCP (multimodal analysis)
    │  └→ GenerationAgent → GenerationMCP (LLM insights)
    │
    ├─ REPORT:
    │  ├→ MultimodalPipeline (transcription + vision)
    │  ├→ GenerationAgent (insights)
    │  └→ ReportAgent (PDF/PPT generation)
    │
    └─ COMBINED:
       ├→ Vision → Generation → Report pipeline
       └→ Return full analysis + report

QueryResponse returned to frontend with:
  - agent name
  - action taken
  - response text
  - confidence
  - transcription + metadata
  - file_path (if PDF/PPT generated)
```

## Key Design Decisions (Phases 1-3)

1. **Intent Classification**: Keyword-based (deterministic, no LLM required)
   - Ensures fast routing even without LLM service
   - 5 intent types cover all use cases
   
2. **gRPC Services**: Three independent services (not monolithic)
   - Decouples concerns
   - Each service can be scaled/restarted independently
   - Enables future MCP tool registration per service
   
3. **MultimodalPipeline**: Centralized structured output
   - All intermediates follow consistent JSON schema
   - Enables composition of agent pipelines
   - Cache in context.metadata for reuse
   
4. **Heuristic Fallback**: No hard dependency on LLM
   - GenerationAgent works even without Llama.cpp
   - Ensures robustness during Phase 4 development
   - GRPCServiceManager verifies services on startup
   
5. **Startup/Shutdown Hooks**: Graceful service lifecycle
   - gRPC services spawned as daemon threads
   - Cleanup on application shutdown
   - Health checks on /status endpoint
